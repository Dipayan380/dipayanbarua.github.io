Microsoft Entra ID

The cloud-native identity platform that replaced Azure Active Directory. This is where every authentication decision starts and where every access policy lives.

What is Entra ID

Microsoft Entra ID (formerly Azure Active Directory) is Microsoft's cloud-based identity and access management service. If you've worked with on-premises Active Directory, think of Entra ID as the cloud equivalent — but it's actually a fundamentally different architecture built for the internet, not for a corporate LAN.

Every Microsoft 365 tenant, every Azure subscription, and every modern Microsoft service relies on Entra ID for authentication. When a user signs into Outlook, Teams, SharePoint, or an Azure portal, Entra ID is the service that verifies who they are and decides what they can access.

ℹ
Key distinction Entra ID is not "Active Directory in the cloud." AD DS uses Kerberos and LDAP inside a domain/forest. Entra ID uses OAuth 2.0, OpenID Connect, and SAML over HTTPS. They solve the same problem (identity) but with completely different protocols for completely different networks.

The core job of Entra ID is answering two questions for every request:

  1. Authentication — is this person who they claim to be? (Verify identity via password, MFA, passwordless, etc.)
  2. Authorization — is this verified person allowed to access this specific resource? (Check roles, group memberships, Conditional Access policies, app permissions)

Tenants and Directories

A tenant is a dedicated instance of Entra ID that an organisation gets when they sign up for any Microsoft cloud service. Think of it as your organisation's private identity boundary in the cloud. It has a globally unique ID (tenant ID) and at least one verified domain name.

Every tenant is completely isolated from every other tenant. Users in Tenant A cannot see or access resources in Tenant B unless explicitly invited via B2B collaboration. This isolation is fundamental to the security model.

Key things to understand about tenants:

  • Each tenant has a default domain: yourorg.onmicrosoft.com
  • You can add and verify custom domains (e.g. compnow.com.au)
  • A tenant can have multiple subscriptions (Azure) and licences (M365) attached to it
  • The tenant is the trust boundary — Conditional Access policies, RBAC, app registrations all live at the tenant level
💡
Lab tip You can create a free Entra ID tenant at entra.microsoft.com to practise. A free tenant gives you up to 50,000 objects and is perfect for learning Conditional Access, RBAC, and app registrations without affecting a production environment.

Users, Groups and Licensing

Users are the core identity objects. Each user has a UPN (User Principal Name, usually their email), an Object ID (a GUID that never changes), and a set of attributes — display name, job title, department, usage location, and more.

Users can be:

  • Cloud-only — created and managed entirely in Entra ID
  • Synced — originating from on-premises AD and synchronised via Entra ID Connect
  • Guest (B2B) — external users invited from another tenant or identity provider

Groups are how you manage access at scale. Instead of assigning permissions to individual users, you assign them to groups and manage membership. Entra ID supports two group types:

  • Security groups — used for access control (RBAC roles, app assignments, Conditional Access targeting)
  • Microsoft 365 groups — used for collaboration (Teams, SharePoint sites, shared mailboxes)

Groups can have assigned membership (you manually add/remove users) or dynamic membership (users are automatically added/removed based on attribute rules). Dynamic groups are extremely powerful for automation — for example, a rule like user.department -eq "Engineering" automatically adds every engineer to the group.

Licensing in Entra ID is tied to users, not devices. You assign licence plans (E3, E5, F1, etc.) to users either directly or via group-based licensing. Group-based licensing is the recommended approach at scale — assign a licence to a group, and any user who joins that group automatically gets the licence.

      
PowerShell

      # Get all users with no licence assigned
Get-MgUser -Filter "assignedLicenses/`$count eq 0" -ConsistencyLevel eventual -CountVariable count -All |
  Select-Object DisplayName, UserPrincipalName, AccountEnabled |
  Sort-Object DisplayName

Conditional Access Policies

Conditional Access is the policy engine at the heart of Entra ID's security model. It evaluates every sign-in against a set of conditions and decides whether to allow, block, or require additional verification.

The logic follows an if/then pattern:

  • IF a set of conditions is true (who is signing in, from where, to what app, on what device, at what risk level)...
  • THEN enforce a set of controls (require MFA, require a compliant device, block access, limit session duration)

The conditions you can evaluate (called signals):

SignalWhat it checksExample
User / GroupWho is authenticatingAll users except break-glass accounts
Cloud appWhat they're accessingExchange Online, Azure Portal
LocationWhere the request comes fromOutside Australia = require MFA
Device platformOS of the deviceiOS, Android, Windows
Device stateCompliance / join statusMust be Intune-compliant
Sign-in riskIdentity Protection risk levelHigh risk = block, medium = MFA
Client appHow they're connectingBrowser vs desktop app vs legacy auth
⚠
Never lock yourself out Always create at least two break-glass (emergency access) accounts that are excluded from ALL Conditional Access policies. These accounts should have strong unique passwords, no MFA (or FIDO2 keys stored securely), and Global Administrator role. Test them regularly. 
      
PowerShell

      # List all Conditional Access policies and their state
Get-MgIdentityConditionalAccessPolicy -All |
  Select-Object DisplayName, State, CreatedDateTime |
  Sort-Object State, DisplayName |
  Format-Table -AutoSize

Multi-Factor Authentication

MFA requires users to prove their identity with at least two different factors: something they know (password), something they have (phone, security key), or something they are (biometrics). It's the single most effective defence against credential-based attacks — Microsoft's own data shows MFA blocks over 99.9% of account compromise attacks.

MFA methods available in Entra ID, from strongest to weakest:

  1. FIDO2 security keys — hardware tokens, phishing-resistant, the gold standard
  2. Windows Hello for Business — biometric or PIN tied to the device's TPM
  3. Microsoft Authenticator (passwordless) — number matching push notifications
  4. Microsoft Authenticator (push) — approve/deny prompts
  5. OATH TOTP tokens — 6-digit codes from any authenticator app
  6. SMS / Voice — one-time codes via text or call (weakest, avoid if possible)
💡
Best practice Enforce MFA via Conditional Access policies, not via per-user MFA settings (the legacy approach). Conditional Access gives you granular control — you can require MFA only for specific apps, locations, or risk levels, rather than forcing it on every single sign-in.

RBAC and Role Assignments

Role-Based Access Control in Entra ID follows the principle of least privilege — give people only the permissions they need, nothing more. Entra ID has ~80+ built-in directory roles, but in practice you'll work with a handful:

RoleWhat it can doWhen to assign
Global AdministratorEverything across the entire tenantBreak-glass accounts only — never for daily use
User AdministratorCreate/manage users and groups, reset passwordsHelpdesk leads, identity team
Intune AdministratorFull Intune managementEndpoint management team
Exchange AdministratorExchange Online managementMessaging team
Security ReaderRead-only security dataSOC analysts, auditors
Global ReaderRead everything, change nothingReporting, compliance, new admins learning

Privileged Identity Management (PIM) takes RBAC further by making role assignments time-bound and approval-based. Instead of permanently assigning someone the Global Admin role, PIM lets them "activate" the role for a limited window (e.g. 4 hours) with justification and optional approval. This dramatically reduces the attack surface of privileged accounts.

App Registrations and SSO

An app registration is how you integrate an application with Entra ID for authentication. When you register an app, Entra ID gives it a unique Application ID and you configure how it authenticates users — which protocol (SAML, OAuth 2.0, OpenID Connect), what permissions it needs, and what tokens it receives.

The two sides of an app in Entra ID:

  • App registration — the global definition of the app (lives in your home tenant, defines the app's identity)
  • Enterprise application (service principal) — the local instance in your tenant that controls who can access it, SSO settings, and provisioning

Single Sign-On (SSO) means a user authenticates once with Entra ID and can then access all integrated apps without signing in again. The three SSO protocols you'll encounter:

  • SAML 2.0 — the most common for enterprise SaaS apps (Salesforce, ServiceNow, Zscaler). You exchange XML-based assertions.
  • OpenID Connect (OIDC) — modern, token-based, built on top of OAuth 2.0. Used by newer apps and SPAs.
  • Password-based SSO — Entra ID stores and auto-fills credentials. A stopgap for apps that don't support federated SSO.
      
PowerShell

      # List all enterprise apps with SAML SSO configured
Get-MgServicePrincipal -Filter "preferredSingleSignOnMode eq 'saml'" -All |
  Select-Object DisplayName, AppId, PreferredSingleSignOnMode |
  Sort-Object DisplayName

Access Reviews and Governance

Access Reviews are a governance feature that lets you periodically audit who has access to what — and revoke access that's no longer needed. They're essential for compliance (SOX, ISO 27001) and for reducing identity sprawl.

You can create access reviews for:

  • Group memberships — "Do all 47 people in this group still need to be there?"
  • App assignments — "Should these 12 people still have access to ServiceNow?"
  • Role assignments — "Are all 3 Global Admins still justified?"
  • Access packages (Entitlement Management) — bundled access that expires

Reviews can be one-time or recurring (monthly, quarterly). Reviewers can be the users themselves ("Do you still need this access?"), their managers, group owners, or specific people you designate. When a reviewer denies access, the user is automatically removed.

Entra ID Connect

Entra ID Connect (formerly Azure AD Connect) is the bridge between on-premises Active Directory and Entra ID. It synchronises user accounts, group memberships, and (optionally) password hashes from your on-prem AD forest to Entra ID, giving users a single identity that works both on the corporate network and in the cloud.

The three authentication methods you can configure with Entra Connect:

  • Password Hash Sync (PHS) — the simplest; hashes of on-prem passwords are synced to Entra ID. Users authenticate directly against Entra ID using the same password they use on-prem. Microsoft's recommended default.
  • Pass-Through Authentication (PTA) — authentication requests to Entra ID are forwarded to an on-prem agent that validates the password against AD in real time. Passwords never leave on-prem.
  • Federation (AD FS) — all authentication is handled by on-premises AD FS servers. The most complex, used when you need advanced claims rules or third-party identity providers.
ℹ
Sync direction matters Entra Connect syncs from on-prem AD to Entra ID. For synced users, the "source of authority" remains on-premises — you manage those users in AD, and changes flow up to the cloud. Cloud-only users are managed directly in Entra ID.

PowerShell for Entra ID

The modern way to manage Entra ID programmatically is via the Microsoft Graph PowerShell SDK. The old AzureAD and MSOnline modules are deprecated — everything is moving to Microsoft.Graph.

      
PowerShell

      # Install the Graph module (one-time)
Install-Module Microsoft.Graph -Scope CurrentUser

# Connect with the scopes you need
Connect-MgGraph -Scopes "User.Read.All","Group.Read.All","Policy.Read.All"

# You're now authenticated — run queries
Get-MgUser -Top 10 | Select DisplayName, UserPrincipalName

Common tasks you'll automate:

  • Bulk user creation — import from CSV using New-MgUser
  • Group membership reports — Get-MgGroupMember piped to export
  • Licence assignment reports — find unlicensed users, over-assigned SKUs
  • Conditional Access audit — list all policies, their state, and targeted groups
  • Stale account cleanup — find users who haven't signed in for 90+ days
      
PowerShell

      # Find users who haven't signed in for 90 days
$cutoff = (Get-Date).AddDays(-90).ToString("yyyy-MM-ddTHH:mm:ssZ")
Get-MgUser -Filter "signInActivity/lastSignInDateTime le $cutoff" -Property DisplayName,UserPrincipalName,SignInActivity -All |
  Select-Object DisplayName, UserPrincipalName, @{N='LastSignIn';E={$_.SignInActivity.LastSignInDateTime}} |
  Sort-Object LastSignIn
💡
Graph Explorer Before writing PowerShell, test your queries in Graph Explorer — it shows you the exact API call, response format, and required permissions. Then translate to PowerShell.
